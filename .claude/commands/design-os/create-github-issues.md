# Create GitHub Issues

You are helping the user create GitHub issues from the generated `product/issues/issues.json` file.

This command assumes:
- The GitHub CLI (`gh`) is authenticated and ready to use
- Issues have been generated via `/generate-issues`
- The planning repository has been pushed to GitHub (for screenshot URLs)

## Step 1: Load and Validate Issues

Read `product/issues/issues.json` and validate:

1. The file exists and is valid JSON
2. `_meta.targetRepo` is set (e.g., `artisan-build/myapp`)
3. `_meta.planningRepo` is set (for screenshot URLs)
4. There are groups with issues to create

**If issues.json is missing:**

"No issues found. Please run `/generate-issues` first to generate issues from your planning documents."

Stop here.

**If targetRepo is not set:**

"The target repository is not configured in issues.json. Please run `/generate-issues` again and provide the target repository when prompted."

Stop here.

## Step 2: Check for Already-Created Issues

Scan all issues and report:

- Total issues: [count]
- Already created: [count] (have githubIssueNumber set)
- To be created: [count]

**If all issues are already created:**

"All [count] issues have already been created on GitHub. Nothing to do.

To recreate issues, you can either:
1. Edit `product/issues/issues.json` and set `githubIssueNumber: null` for issues you want to recreate
2. Run `/generate-issues` again to regenerate all issues"

Stop here.

**If some issues exist:**

"Found [total] issues:
- [created] already created on GitHub
- [remaining] to be created

Would you like to continue creating the remaining [remaining] issues?"

Use AskUserQuestion with options:
- "Yes, create remaining issues"
- "No, stop here"

## Step 3: Confirm Before Creating

Present a summary before creating:

"Ready to create [count] issues on `[targetRepo]`:

**Group 1: [name]** ([count] issues)
- [issue title 1]
- [issue title 2]

**Group 2: [name]** ([count] issues)

Issues will be created in dependency order. Each issue body will include:
- Description
- Tasks as checkboxes
- Acceptance criteria
- Dependencies (linked to GitHub issue numbers)
- Screenshots (embedded from planning repo)

Proceed with creating these issues?"

Use AskUserQuestion with options:
- "Yes, create all issues"
- "No, cancel"

## Step 4: Create Issues in Order

Create issues group by group, respecting dependencies within groups.

For each issue, format the body as:

```markdown
## Description

[issue.description]

## Tasks

- [ ] [task 1]
- [ ] [task 2]

## Acceptance Criteria

- [criterion 1]
- [criterion 2]

## Dependencies

Blocked by:
- #[issue number] - [issue title]

## Screenshots

![Screenshot](https://raw.githubusercontent.com/[planningRepo]/main/[screenshot path])

---
*Generated by Design OS*
```

**Creating each issue:**

Use the GitHub CLI:

```bash
gh issue create \
  --repo "[targetRepo]" \
  --title "[issue.title]" \
  --label "[label1],[label2]" \
  --body "[formatted body]"
```

**Important notes:**

1. **Create labels first** — Before creating the first issue, ensure labels exist:
   ```bash
   gh label create foundation --repo "[targetRepo]" --color "8B5CF6" --description "Core infrastructure" --force
   gh label create feature --repo "[targetRepo]" --color "3B82F6" --description "Feature implementation" --force
   gh label create ui --repo "[targetRepo]" --color "10B981" --description "UI/UX work" --force
   gh label create infrastructure --repo "[targetRepo]" --color "F59E0B" --description "DevOps and infrastructure" --force
   gh label create human-task --repo "[targetRepo]" --color "F43F5E" --description "Requires manual human action" --force
   gh label create polish --repo "[targetRepo]" --color "78716C" --description "Polish and refinement" --force
   ```

2. **Capture issue numbers** — After each `gh issue create`, capture the returned issue number

3. **Update dependencies** — When an issue has `dependsOn` references, look up the `githubIssueNumber` from previously created issues and include them in the body

4. **Handle screenshots** — Include screenshot images using the raw GitHub URL format

5. **Progress updates** — After creating each issue, report progress:
   "Created issue #[number]: [title]"

## Step 5: Update issues.json

After creating each issue, update `product/issues/issues.json` to set the `githubIssueNumber` field.

This is critical so that:
- The UI shows which issues have been created
- Dependencies can reference actual GitHub issue numbers
- Re-running the command skips already-created issues

**Update the file after each issue** (not at the end) to preserve progress if the command is interrupted.

## Git: Commit and Push

After creating issues, commit the updated issues.json:

```bash
git add product/issues/issues.json && git commit -m "Update issues.json with GitHub issue numbers" && git push
```

## Error Handling

**If gh CLI fails:**

"Failed to create issue '[title]': [error message]

This might be due to:
- Repository doesn't exist or you don't have access
- Network issues
- Rate limiting

Would you like to:
1. Retry this issue
2. Skip this issue and continue
3. Stop and fix the problem"

**If label creation fails:**

Labels are created with `--force` so they should update if they exist. If creation fails entirely, continue without labels and note the issue.

**If a dependency hasn't been created yet:**

Issues should be created in order, but if a dependency is missing:
- Note the dependency by name instead of number
- Continue creating the issue
- Flag it for manual linking later

## Important Notes

- Always create issues in group order (Group 1 before Group 2, etc.)
- Within a group, create issues in array order (this should already respect dependencies)
- The `githubIssueNumber` field is the source of truth for whether an issue exists
- Screenshot URLs use the raw.githubusercontent.com format
- Keep the issues.json updated incrementally for resilience
- Labels are created with specific colors matching the Design OS UI
